# 01_역량검증가이드: 무결점 Table & Form Parser 개발 가능성 진단

회비서님, 다온입니다. 

기존에 간단한 RAG를 구현해보신 경험은 AI의 텍스트 처리와 프이프라인 구성의 기초를 이해하고 있다는 점에서 훌륭한 자산입니다. 하지만 PRD에 명시된 **'100% 로컬 망분리 기반의 99.9% 무결점 표 파싱 및 Confidence Score 산출'** 기능은 일반적인 텍스트 기반 RAG와는 기술적 난이도와 결이 다릅니다.

회비서님께서 AI 코딩 어시스턴트의 도움을 받아 이 기능을 개발할 수 있을지 객관적으로 점검하기 위한 기준과, 이를 당장 테스트해 볼 수 있는 3단계 체크리스트를 정리했습니다.

## 1. 현재 역량(RAG 경험)과 목표(파서)의 기술적 차이 수용
| 구분 | 간단한 RAG 개발 (현재) | 무결점 Table Parser 개발 (목표/PRD 기준) | 필요한 추가 학습/접근법 |
| :--- | :--- | :--- | :--- |
| **데이터 형태** | 1차원 Text 위주 (단순 텍스트 청킹/추출) | 2차원 공간 데이터 (표의 행렬, 셀 병합, Bounding Box) | OCR, Document Layout Analysis, Vision AI에 대한 이해 |
| **활용 기반** | 상용 API (OpenAI 등 클라우드 의존) | **100% 로컬 구동 (폐쇄망 완전 준수)** | 오픈소스 LLM/VLM 구동 (Ollama, vLLM), 모델 경량화 및 서버 세팅 (Docker) |
| **결과 검증** | 프롬프트 결과 수용 (환각 발생 가능성 존재) | **Confidence Score 산출 및 의심 영역 하이라이트** | Logprob 데이터 분석 또는 LLM 교차 검증 기반 점수화(Scoring) 로직 |
| **주요 로직** | Embedding -> Vector DB -> 검색 및 생성 | **Rule-based (로펌/회사 고정 템플릿) + AI (구조화 방어) 융합** | 예외 처리 하드코딩, 정규식, 휴리스틱과 AI를 혼합하는 아키텍처 설계 능력 |

## 2. 개발 가능성 자가 진단 방법론 (PoC 테스트)

역량이 다소 부족하더라도, AI 기술 블록들을 결합하고 조율(Orchestration)하는 능력만 있다면 충분히 해낼 수 있습니다. 내가 AI의 지도를 받으며 파이프라인을 통제할 수 있을지 다음 3단계 미션을 통해 스스로 테스트(PoC)해보시길 권합니다.

### [Phase 1] 구조화 파일 파싱 직관성 확인 (클라우드 환경 1일)
- **목표:** 병합된 셀이 복잡하게 얽힌 로펌 계약서 혹은 양식 PDF 1장을 API를 써서 Markdown이나 JSON으로 구조 파괴 없이 파싱하기.
- **도구:** LlamaParse, Unstructured.io 등의 API.
- **역량 체크포인트:** 단순 추출을 넘어, 일부분 셀이 밀리거나 누락되는 엣지 케이스(Edge case)가 터졌을 때 AI의 도움을 받아 파이썬 스크립트나 구체적인 프롬프트 제어로 이를 복구해낼 수 있는가?

### [Phase 2] 신뢰도(Confidence Score) 추출 처리 (1~2일)
- **목표:** 파싱 결과물을 뱉어낼 때, 각 구문/셀 데이터의 '확신도 확률(0~100점)'을 배열에 함께 담기 (PRD의 붉은색 하이라이팅 기준점).
- **도구:** LLM API의 Logprobs 파라미터 활용 또는 Self-correction 프롬프팅 구조.
- **역량 체크포인트:** 응답 텍스트만 빼내는 수준을 넘어, 모델이 반환하는 메타데이터(확률값)를 파싱하고 기준값(80점) 미만인 구역 좌표를 배열로 정리하는 코드를 AI에게 요구하고 디버깅할 수 있는가?

### [Phase 3] 폐쇄망 로컬 환경 배포 (가장 큰 장벽 / 2~3일)
- **목표:** 인터넷을 차단한 상태에서 오픈소스 파서 모델을 띄워 Phase 1 기능 동일하게 실행하기.
- **도구:** Ollama/vLLM, Tesseract OCR / Table Transformers 등 로컬 오픈소스 비전/언어모델.
- **역량 체크포인트:** 하드웨어 가속기(GPU/CUDA) 설정, 메모리 에러(OOM), 라이브러리 충돌 시 AI와 토론해가며 로컬 서버에 도커(Docker) 기반 스택을 무사히 세팅할 수 있는가?

## 3. 다온의 현실적 판단 및 제안

회비서님, **99.9% 무결점 로컬 파싱 엔진을 처음부터 끝까지(From Scratch) 혼자 짜는 것은 현재 역량 대비 매우 무리한 목표일 수 있습니다.** 

하지만, 최근의 오픈소스 생태계(LlamaParse 로컬 구축 가이드 등)와 **AI 어시스턴트를 활용하여 "강력한 오픈소스들을 결합하여 우리 비즈니스 로직(Rule-based 필터링) 맞게 파이프라인을 연결하는 일"이라면 해볼 만한 도전입니다.**

무엇보다 당장 1일 정도 시간을 내어 **Phase 1과 Phase 2를 Colab이나 로컬 Python 환경에서 직접 부딪혀 보십시오.** 수많은 에러 로그를 만났을 때, AI가 제시하는 디버깅 코드를 어떻게 이해하고 적용할지가 곧 회비서님의 개발 역량 한계선을 결정할 것입니다. 

테스트를 진행하시기로 결정하신다면, 첫 진입을 위한 아키텍처 스케치나 구체적인 PoC 스크립트는 제가 작성해 드리겠습니다. 언제든 말씀해주십시오.
