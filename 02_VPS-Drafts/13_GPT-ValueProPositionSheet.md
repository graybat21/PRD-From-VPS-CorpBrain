작성 방식은 `8 ValueProposion_Sheet 작성방법`의 구조를 따르되, 페르소나·JTBD·시장 규모·경쟁사·고객 여정을 한 장에서 연결되도록 구성했습니다. 전통적인 브레인스토밍형 캔버스보다, 분석 결과를 바로 사업 문장으로 바꾸는 통합 문서가 더 적합하다는 방향을 따랐습니다.

# Value Proposition Sheet

## 1) 사업 한 줄 정의

**CorpBrain은 문서와 파일이 파편화된 SME가 핵심 지식과 업무 데이터를 잃지 않도록, 무마찰 연동과 무결점 파싱, 검수 보조 UI를 통해 비정형 데이터를 신뢰 가능한 자산으로 바꿔주는 실시간 데이터 클리닝 OS이다.**
이 정의는 “고객이 누구인지, 어떤 문제를 얼마나 심하게 겪는지, 우리 솔루션이 무엇을 바꾸는지”를 한 문장에 넣는 방식입니다. 핵심 타깃은 Q1에 위치한 시리즈 B CTO와 부티크 특화 로펌 대표이며, 확장 타깃으로 회계법인이 설정되어 있습니다.

## 2) 타깃 고객

| 구분         | 핵심 대상                       | 왜 이 고객인가                                                            |
| ---------- | --------------------------- | ------------------------------------------------------------------- |
| Primary 1  | **부티크 특화 로펌 대표 / 파트너 변호사**  | 표·특수양식 파싱 실패가 곧 법적 리스크로 이어지고, 보안 민감도와 지불 의사가 가장 높습니다. DOS 최상위입니다.   |
| Primary 2  | **시리즈 B R&D 스타트업 CTO**      | 레거시 지식 단절, 퇴사로 인한 블랙박스화, 툴 분절 문제가 직접적인 매출 손실로 연결됩니다. 도입 속도도 빠릅니다.   |
| Secondary  | **회계법인 파트너**                | 시즌성 문서 대량 처리와 ERP 연동 수요가 강해, Core 검증 후 확장하기 좋은 세그먼트입니다.             |
| Non-target | **대형 금융권 / B2C / 단순 챗봇 수요** | 망분리 강박, 낮은 WTP, 범용 챗봇 대체 가능성으로 초기 사업 타깃에서 제외하는 것이 맞습니다.             |

## 3) 고객이 겪는 핵심 문제

| 고객군  | Pain                         | 현 상태의 문제                                                        |
| ---- | ---------------------------- | --------------------------------------------------------------- |
| 로펌   | 표와 특수 양식이 깨지면 의사결정 자체가 위험해짐  | 주니어가 수작업으로 4시간 이상 검수·재입력하며, 실수 한 번이 치명적입니다.                     |
| CTO  | 퇴사·온보딩 때 문서와 지식이 파편화되어 블랙박스화 | Slack, GitHub, Notion이 분절되어 최신 문서를 가려내기 어렵고, 잘못된 버전이 사고를 냅니다.   |
| 회계법인 | 영수증·장부·증빙의 양식이 너무 다양함        | 시즌마다 반복 작업이 폭증하고, 기존 OCR/RPA만으로는 실무 검수가 끝나지 않습니다.               |

## 4) JTBD 관점의 핵심 Job

| 고객군  | Job Statement                                                       |
| ---- | ------------------------------------------------------------------- |
| 로펌   | **단 1%의 구조 붕괴도 없이 특수 양식을 정형화하고, 보안 리스크 없이 바로 검토 가능한 데이터로 바꾸고 싶다.**  |
| CTO  | **새 툴 학습을 강요하지 않으면서, 흩어진 지식을 최신 상태로 통합해 온보딩과 유지보수를 빠르게 만들고 싶다.**    |
| 회계법인 | **양식이 제각각인 증빙을 빠르게 구조화해, 결산과 신고 마감을 줄이고 싶다.**                       |

## 5) 고객이 원하는 Outcome

| Outcome          | 기대 변화                                              |
| ---------------- | -------------------------------------------------- |
| 수동 검수 시간 감소      | 로펌 기준 4시간 검수 작업을 30분 수준으로 줄이는 것이 핵심 기대치입니다.        |
| 특수 양식 인식 정확도 향상  | 표/도표/조항의 구조가 깨지지 않아야 합니다. 이것이 구매 트리거입니다.           |
| 온보딩·마이그레이션 시간 단축 | Days → Hours 수준으로 줄이는 것이 CTO 세그먼트의 핵심 Outcome입니다.  |
| 검수 신뢰도 확보        | Confidence Score 기반으로 위험 구간만 검토하게 만드는 경험이 중요합니다.   |

## 6) 우리 솔루션의 핵심 가치 제안

| 항목    | 내용                                                                                               |
| ----- | ------------------------------------------------------------------------------------------------ |
| 핵심 제안 | **무마찰 연동 + 무결점 파싱 + 검수 보조 UI**로, 고객이 기존 툴을 바꾸지 않아도 데이터가 자산이 되게 만든다.                              |
| 해결 방식 | 1) 기존 툴에서 증분 데이터만 가져오기, 2) 구버전/중복/쓰레기 데이터 제외, 3) 구조화 결과를 신뢰도 기반으로 하이라이트, 4) 필요 시 온프레미스/VPC 고려.   |
| 고객 효익 | 사람의 수작업을 줄이고, 잘못된 인식으로 인한 리스크를 낮추며, 도입 장벽을 낮춘다.                                                  |

## 7) 기존 대안(Competitor / Substitute)

| 대안                                          | 한계                                                      |
| ------------------------------------------- | ------------------------------------------------------- |
| 범용 OCR / 문서 파서                              | 표·차트·특수 서식을 깨뜨리기 쉽고, 전문직 실무에서 바로 쓰기 어렵습니다.              |
| 수동 엑셀 정리                                    | 정확성은 높일 수 있지만, 비용과 시간이 너무 큽니다.                          |
| 글로벌 전처리 플랫폼(Unstructured.io, LlamaIndex)    | 인프라·개발자 친화적이지만, SME의 레거시 파일 정리와 도입 마찰 해소에는 직접적이지 않습니다.  |
| 버티컬 엔터프라이즈 솔루션(Upstage, Allganize, GridOne) | 대기업·규제 산업 중심으로 강하지만, 예비창업자 관점의 SME 초기 진입에는 무겁습니다.       |

## 12) 최종 요약 문장

**이 사업의 본질은 “문서를 잘 읽는 기술”이 아니라, “고객이 믿고 바로 업무에 쓸 수 있는 자산으로 바꾸는 능력”입니다.**
초기에는 **로펌의 특수 양식 파싱**과 **CTO의 레거시 지식 정리**를 양대 축으로 잡고, 그 다음 **회계법인**으로 확장하는 구도가 가장 자연스럽습니다. 이 선택은 시장 규모, 진입 난이도, 지불 의사, 경쟁 지형을 모두 함께 고려한 결과입니다.
